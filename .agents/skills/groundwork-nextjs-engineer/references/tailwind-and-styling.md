# Tailwind & Theming

This reference covers **how** to wire and consume the design system in Tailwind v4 — the mechanics, not the values. Colours, type, spacing, radii, elevation, blur, gradients, surfaces, and motion are per-app decisions projected from `.groundwork/config/brand-tokens.json` into `app/brand.css`; see `references/visual-language.md` for the projection chain. Never bake values into the skill or a component.

## Tailwind v4 CSS-First Architecture

Tailwind v4 eliminates the JavaScript configuration file. All customisation happens in CSS via `@theme` blocks and CSS custom properties. There is no `tailwind.config.ts` in the Next.js application.

| v3 (Forbidden) | v4 (Required) |
|----------------|---------------|
| `tailwind.config.ts` | No config file — use `@theme` in CSS |
| `@apply` with `theme()` | Direct CSS custom properties |
| `content: ['./app/**/*.tsx']` | Automatic content detection |
| `darkMode: 'class'` | `@custom-variant dark (&:is(.dark *))` |
| `plugins: [require('...')]` | `@plugin` directive in CSS |

## globals.css and brand.css structure

Styling is split into a **generated value file** and a **static structure file**:

- **`app/brand.css`** — GENERATED by the nextjs-app generator from the design system's `visual` block. It carries the shadcn structural variables (`--background`, `--primary`, …) and the `--gw-*` atmosphere values (blur, elevation stacks, gradients, surface tints, type-role micro, motion profiles), for both light (`:root`) and dark (`.dark`). **Never hand-edit it** — re-run the design system and regenerate.
- **`app/globals.css`** — STATIC structure. It imports Tailwind, `tw-animate-css`, and `brand.css`; declares the `dark` variant; maps the variables into Tailwind token utilities via `@theme inline`; and defines the surface utility classes (`.surface-glass`, `.surface-elevated`, `.surface-hero`) that compose the `--gw-*` tokens. Extend its structure if a project needs a new mapping or surface class — never bake a literal value into it.

This is the same generated-theme pattern electron-app and flutter-app use. The values change per app; the structure does not.

## PostCSS Pipeline

Tailwind v4 uses `@tailwindcss/postcss`:

```js
// postcss.config.js
module.exports = {
  plugins: {
    '@tailwindcss/postcss': {},
  },
};
```

Do not add `autoprefixer` separately — `@tailwindcss/postcss` handles vendor prefixing.

## Dark Mode with @custom-variant

Class-based dark mode: the `dark` variant activates when `.dark` is on an ancestor.

```css
@custom-variant dark (&:is(.dark *));
```

Because the tokens themselves flip in the `.dark` scope (brand.css redefines every structural variable for dark), most components need **no** `dark:` variants — consuming a token utility switches automatically:

```tsx
<div className="bg-background text-foreground">Auto-switches with the theme</div>
```

Reach for a `dark:` variant only when the two themes need structurally different styles (a different border, shadow, or opacity), not merely a different colour.

## Consuming tokens as Tailwind utilities

`@theme inline` in `globals.css` maps each variable to a Tailwind utility. Consume the utility, in priority order:

1. **Token utilities** — `bg-background`, `text-foreground`, `bg-primary`, `text-muted-foreground`, `border-border`, `shadow-low/mid/high`, `backdrop-blur-subtle/standard/heavy`, the spacing scale (`p-4`, `gap-6`), the radius scale (`rounded-lg`). This is the default path.
2. **Surface classes** — `.surface-glass` / `.surface-elevated` / `.surface-hero` for composed depth (blur + tint + border + shadow + gradient).
3. **CSS-variable reference** — for a projected value with no dedicated utility, use the var-reference form `shadow-(--gw-shadow-mid)` or `bg-(--gw-surface-glass-tint)`.

**Never use arbitrary literals.** `bg-[#3b82f6]`, `p-[12px]`, `shadow-[0_1px_2px…]`, `backdrop-blur-[20px]`, and inline gradient strings bypass the design system and **fail the token-conformance lint** (`eslint.config.mjs`). The var-reference form (`(--gw-…)`) is allowed; the literal form (`[…]`) is not.

## Animations with tw-animate-css

`tw-animate-css` provides utility-based enter/exit animations without custom keyframes:

```tsx
<div className="animate-in fade-in duration-200">Appears smoothly</div>
<DialogContent className="animate-in fade-in slide-in-from-bottom-4 duration-300 data-[state=closed]:animate-out data-[state=closed]:fade-out">
  Dialog content
</DialogContent>
```

### Motion mechanics

The durations, easing, and transforms are the design system's call, projected as `--gw-motion-<context>-{duration,ease,transform}` (hover, press, enter, exit, stagger). Consume those rather than inventing timings. Three mechanics are universal and always apply:

- **Animate compositor-only properties** — `transform` and `opacity` hold frame rate; animating `width`/`height`/`box-shadow`/`top` drops frames.
- **Respect `prefers-reduced-motion`** — gate non-essential motion behind the reduced-motion query.
- **Entry eases out, never linear on interface elements** — the specific curve comes from the motion tokens.

## Theming and dual-theme verification

The app ships a light and a dark theme; a `ThemeProvider` toggles the `.dark` class on `<html>` (persisted to `localStorage`, honouring the system preference on first load). The theme *names* and *values* belong to the design system — keep the provider generic over "light" and "dark", never hardcode brand-specific theme identities here.

**Every component is verified in both themes before merging.** Watch for:

- Text contrast meeting WCAG AA (4.5:1 body, 3:1 large text) in both themes.
- Shadow visibility — a stack tuned for light can vanish on dark surfaces (this is why brand.css carries deeper dark-theme elevation).
- Border visibility — translucent hairline borders can disappear in light mode.
- Backdrop-blur surfaces rendering correctly against both backgrounds.
- Semantic colours (success/error/warning) staying readable in both modes.

## Common Mistakes

| Mistake | Why It Fails | Fix |
|---------|-------------|-----|
| Creating `tailwind.config.ts` | Overrides CSS-first architecture | Use `@theme` in `globals.css` |
| Hand-editing `brand.css` | Overwritten on regeneration | Change the design system and regenerate |
| Using `theme()` in CSS | v3 syntax, unsupported | Reference CSS custom properties directly |
| Arbitrary literal (`bg-[#…]`, `shadow-[…]`, `backdrop-blur-[…]`) | Bypasses tokens; fails lint | Use a token utility, surface class, or `(--gw-…)` var reference |
| Hardcoding `bg-gray-800` | Ignores the projected palette | Use `bg-background` / `bg-card` / `bg-primary` |
| Adding `autoprefixer` | `@tailwindcss/postcss` handles it | Remove it |
| `darkMode: 'class'` in config | v3 syntax | Use `@custom-variant dark` |
